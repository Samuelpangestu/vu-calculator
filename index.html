Metode Kalkulasi & Formula

Kalkulasi Peak TPS
Peak TPS = (Daily √ó 15%) √∑ 3600
Standar industri: 15% transaksi harian terjadi di jam sibuk.

Kenapa 15%?

  • Berdasarkan observasi industri e-commerce, banking, dan telco: 10-20% traffic harian terkonsentrasi di peak hour (jam sibuk)
  • 15% adalah nilai rata-rata yang digunakan sebagai K-factor (peak hour factor) di traffic engineering
  • E-commerce: peak 12:00-14:00 & 19:00-21:00 (12-18% traffic)
  • Banking: peak 08:00-10:00 (10-15% traffic)
  • Social Media: peak 12:00-13:00 & 21:00-22:00 (15-20% traffic)

Kenapa √∑ 3600?

  • 1 jam (hour) = 3600 detik (60 menit √ó 60 detik)
  • TPS = Transactions Per Second, jadi perlu konversi dari per hour ke per second

Kalkulasi VU (Little's Law)
VUs = Peak TPS √ó Session Duration
Little's Law: N = Œª √ó W. Formula inti untuk kapasitas concurrent user.

Apa itu Little's Law?

Formula matematika dari queueing theory (1961) oleh John Little (MIT): N = Œª √ó W

  • N = Jumlah item dalam sistem (concurrent users)
  • Œª (lambda) = Arrival rate (TPS/RPS)
  • W = Waktu rata-rata dalam sistem (Response Time + Think Time)

Distribusi RPS
RPS = (VU √ó API Calls) √∑ Session Duration
Mendistribusikan aksi user sepanjang durasi session.

Perbedaan TPS vs RPS:

  • TPS (Transactions Per Second): Business transactions (end-user actions)
  • ‚Üí Contoh: 1 checkout, 1 login, 1 search
  • RPS (Requests Per Second): Technical HTTP requests
  • ‚Üí Contoh: 1 checkout = 10 HTTP requests (cart API, payment API, inventory API, etc.)

Kenapa Perlu Hitung RPS?

  • Performance tools (k6, JMeter, Gatling) mengukur RPS, bukan TPS
  • Infrastructure metrics (load balancer, API gateway) melaporkan RPS
  • Contoh: 10 TPS √ó 5 API calls = 50 RPS di monitoring dashboard

Little's Law in Performance Testing
UTAMA
PerfMatrix
Penjelasan komprehensif tentang aplikasi Little's Law dalam performance testing dengan formula extended untuk workload modeling.
N = TPS √ó (RT + TT + Pacing)

Apa itu Little's Law? Little's Law adalah rumus matematika yang menghubungkan: jumlah item dalam sistem (N), kecepatan kedatangan item (Œª), dan waktu rata-rata item di dalam sistem (W).

Analoginya: Bayangkan kasir supermarket. Jika 10 orang datang per jam dan setiap orang butuh 6 menit dilayani, maka rata-rata ada 1 orang dalam antrian.

Dalam Performance Testing:

N = jumlah Virtual Users aktif bersamaan

Œª = TPS (transaksi per detik)

W = total waktu satu transaksi

‚ÜíResponse Time (RT): Waktu server memproses request
‚ÜíThink Time (TT): Waktu user "berpikir" sebelum aksi berikutnya
‚ÜíPacing: Jeda tambahan untuk mengontrol rate request

Throughput in Performance Testing
OVERVIEW
BrowserStack
Penjelasan detail tentang metrik throughput dan cara menentukan maximum throughput.

TPS vs Throughput - Apa Bedanya?

TPS: Jumlah transaksi bisnis per detik (login, checkout)

Throughput: Volume data yang diproses per waktu (KB/s, requests/s)

Cara Menemukan Maximum Throughput:

1.Mulai dengan load rendah
2.Tingkatkan load secara bertahap
3.Pantau response time - ketika naik signifikan, mendekati limit
4.Titik dimana throughput tidak naik lagi = maximum throughput

Analogi: Seperti jalan tol - bisa menampung banyak mobil, tapi setelah kapasitas tertentu mulai macet.

Jenis-jenis Performance Test:

Stress Testing

Menguji sistem di atas kapasitas normal. Tujuan: menemukan breaking point dan melihat bagaimana sistem fail.

Terus menaikkan load sampai sistem gagal. Tujuan: menemukan kapasitas maksimum absolut.

üîß Performance Testing Troubleshooting Guide

Panduan identifikasi dan solusi untuk masalah umum dalam performance testing.

‚ö†Ô∏èCommon Performance Bottlenecks

High Response Time
ISSUE
Response time meningkat drastis saat load tinggi.

Kemungkinan Penyebab:

1. Database Bottleneck

Gejala: Slow queries, connection pool exhausted, high DB CPU

Solusi:

‚ÜíTambah index pada kolom yang sering di-query
‚ÜíOptimize slow queries (use EXPLAIN)
‚ÜíIncrease connection pool size
‚ÜíImplement caching (Redis, Memcached)
‚ÜíConsider database replication (read replicas)
2. Application Server Bottleneck

Gejala: High CPU usage, thread pool exhausted, memory pressure

Solusi:

‚ÜíProfile aplikasi untuk menemukan hotspot
‚ÜíOptimize algoritma dan data structures
‚ÜíIncrease thread pool size
‚ÜíHorizontal scaling (add more app servers)
‚ÜíImplement async processing untuk long tasks
3. Network Bottleneck

Gejala: High latency, packet loss, bandwidth saturation

Solusi:

‚ÜíUse CDN untuk static assets
‚ÜíCompress responses (gzip, brotli)
‚ÜíMinimize payload size
‚ÜíUpgrade network bandwidth
High Error Rate
ISSUE
Error rate meningkat saat load testing (5xx errors, timeouts).

Troubleshooting Steps:

500 Internal Server Error
1.Check application logs untuk exception details
2.Verify database connections tidak habis
3.Check memory usage (possible OutOfMemory)
4.Review recent code changes
502/504 Gateway Timeout
1.Increase timeout settings di load balancer
2.Check upstream server health
3.Verify firewall rules
4.Scale up backend servers
Connection Refused
1.Service might be down - check health endpoint
2.Port may be blocked - verify firewall
3.Max connections reached - increase limit
Memory Leak
ISSUE
Memory usage terus meningkat dan tidak turun (heap exhaustion).

Cara Detect Memory Leak:

‚ÜíMemory usage naik terus selama endurance test
‚ÜíGC tidak berhasil free memory
‚ÜíOutOfMemory errors setelah beberapa jam
‚ÜíResponse time degradasi over time
Investigasi Steps:

Java/JVM:

1.Generate heap dump: jmap -dump:live,format=b,file=heap.bin [PID]
2.Analyze dengan Eclipse MAT atau VisualVM
3.Cari objects yang terus bertambah
4.Trace allocation path

Node.js:

1.Use --inspect flag untuk debugging
2.Chrome DevTools Memory Profiler
3.Clinic.js untuk heap snapshots
Common Causes:
‚ÜíUnclosed database connections
‚ÜíEvent listeners tidak di-cleanup
‚ÜíCaching tanpa eviction policy
‚ÜíStatic collections yang terus bertambah
Throughput Plateau
ISSUE
Throughput tidak naik meskipun VUs ditambah (capacity limit reached).

Identifikasi Bottleneck:

Resource Saturation Check:
‚ÜíCPU: Jika CPU > 80%, tambah cores atau optimize code
‚ÜíMemory: Jika RAM > 90%, tambah memory atau fix memory leak
‚ÜíDisk I/O: Jika iowait high, gunakan SSD atau optimize queries
‚ÜíNetwork: Jika bandwidth saturated, upgrade network
‚ÜíDatabase: Jika DB connections maxed, increase pool size
Scaling Strategy:

Vertical Scaling (Scale Up): Tambah CPU/RAM/Disk pada existing server

Horizontal Scaling (Scale Out): Tambah jumlah servers dengan load balancer

Rekomendasi: Horizontal scaling lebih resilient dan cost-effective untuk high traffic

‚úÖ Performance Testing Checklist

Comprehensive checklist untuk setiap fase performance testing - dari planning sampai reporting.

üìãPhase 1: Pre-Test Planning & Preparation

Requirements Gathering

  • Identify business-critical user journeys
  • Gather production traffic data (peak hours, daily transactions)
  • Define SLA/SLO targets (response time, error rate, throughput)
  • Identify third-party dependencies dan APIs
  • Get stakeholder approval untuk test scenarios

Test Environment Setup

  • Verify test environment mirrors production (hardware, software, config)
  • Ensure adequate test data volume (minimum 10K records per table)
  • Configure monitoring tools (APM, logging, metrics)
  • Setup load generator VMs dengan specs adequate
  • Verify network connectivity antara load gen dan target
  • Whitelist load generator IPs di firewall

Script Development

  • Develop test scripts untuk setiap user journey
  • Implement realistic think times
  • Add correlation untuk dynamic values (session IDs, tokens)
  • Parameterize test data (usernames, search terms, etc)
  • Add assertions untuk validate responses
  • Debug scripts dengan 1 VU untuk verify functionality

Baseline Test

  • Run baseline test dengan 1-5 VUs
  • Document baseline metrics (response time, throughput)
  • Verify zero errors dalam baseline test

‚ñ∂Ô∏èPhase 2: Test Execution & Monitoring

Pre-Execution Checks

  • Notify stakeholders tentang test schedule
  • Verify all systems are healthy sebelum test
  • Clear logs dan reset counters
  • Document test parameters (VUs, duration, ramp-up)

Real-time Monitoring

  • Monitor response time trends
  • Track error rate (should be < 1%)
  • Watch CPU usage pada application servers
  • Monitor memory consumption
  • Check database connection pool usage
  • Verify network bandwidth tidak saturated
  • Monitor load generator health (CPU, memory, network)

Issue Response

  • Document any errors atau anomalies yang terjadi
  • Take screenshots dari monitoring dashboards
  • Capture thread dumps jika ada performance degradation
  • Save application logs untuk analysis

üìäPhase 3: Post-Test Analysis & Reporting

Data Collection

  • Collect test results (response time, throughput, errors)
  • Export monitoring data (APM, server metrics)
  • Archive application logs
  • Save database slow query logs

Analysis

  • Compare results vs SLA/SLO targets
  • Identify performance bottlenecks
  • Analyze error patterns
  • Review resource utilization (CPU, memory, network, disk)
  • Calculate percentiles (p50, p90, p95, p99)
  • Identify slow transactions (> 3 seconds)

Reporting

  • Create executive summary (Pass/Fail, key findings)
  • Document test configuration (VUs, duration, scenarios)
  • Include graphs (response time, throughput, errors)
  • List performance bottlenecks dengan evidence
  • Provide actionable recommendations
  • Prioritize issues (Critical, High, Medium, Low)

Follow-up

  • Present findings ke stakeholders
  • Create tickets untuk identified issues
  • Schedule retest setelah fixes implemented
  • Archive all test artifacts untuk future reference
